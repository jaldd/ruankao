# 02 团队绩效域

> **核对状态**：✅ 已核对（核对日期：2026-05-16）

## 一、一句话理解
团队绩效域就是"带好队伍"——把合适的人凑到一块，让大家有干劲、有能力、有协作，最终把活干漂亮。

## 二、管不好会出什么乱子
1. **核心骨干突然集体离职**：团队氛围差、没成长空间，项目做到一半人跑光了，直接崩盘。
2. **互相甩锅没人担责**：分工不清、责任不明，出问题都说"不是我的锅"，项目推进不下去。

## 三、预期目标与指标对照（核心！教材原文）

> **🧭 三者关系解读**：绩效要点 → 预期目标 → 指标。三者是**路径→终点→尺子**的因果链，不是三个独立的东西。下表的"预期目标"就是第四节"绩效要点"要达成的结果，"指标"就是检测目标达没达成的尺子。
>
> **本域映射**：团队绩效域有3个要点，恰好对应3个目标，是最整齐的绩效域！
> - 要点1(项目团队文化)→**你要打造什么**：打造团队文化，贡献1个目标
> - 要点2(高绩效项目团队)→**你要打造什么**：打造高绩效团队，贡献1个目标
> - 要点3(领导力技能)→**你要练什么**：提升团队领导力技能，贡献1个目标
>
> ```
> 要点1：项目团队文化 ──→ ①共享责任          ←尺子：目标和责任心
> 要点2：高绩效项目团队 ─→ ②建立高绩效团队   ←尺子：信任协作+适应变化+彼此赋能
> 要点3：领导力技能 ────→ ③展现领导力和人际技能 ←尺子：管理和领导力风格适宜性
> ```
>
> **逻辑链白话解读**（每个箭头为什么成立？）：
> - 【你要打造什么：文化】要点1→①共享责任：**文化决定行为**，文化好了大家自然把项目当自己的事，尺子"目标和责任心"=检验大家有没有把成果当自己的事
> - 【你要打造什么：高绩效团队】要点2→②建立高绩效团队：**高绩效不是管出来的，是打造出来的**，9个特征齐了就是高绩效团队，尺子"信任协作+适应变化+彼此赋能"=检验团队有没有这些特征
> - 【你要练什么：领导力】要点3→③展现领导力：**领导力不是PM一个人的事**，每个成员都要有领导力技能，尺子"管理和领导力风格适宜性"=检验风格是否适合项目环境

| 预期目标（教材原文） | 大白话解释 | 指标及检查方法（教材原文） | 做得好 | 搞砸了 |
|---------|-----------|-------------|--------|--------|
| ①共享责任 | 大家是不是"一条心"？所有人都了解项目愿景，都把项目成果当成自己的事 | 目标和责任心 | 团队成员主动补位，有人请假其他人能无缝接手 | "这不是我的职责"挂嘴边，互相推诿 |
| ②建立高绩效团队 | 团队彼此信任、相互协作；能适应变化，遇到挑战有韧性；团队感到被赋能 | 信任与协作程度、适应变化的能力、彼此赋能 | 交付质量高、缺陷少、进度稳定、能持续改进 | bug满天飞、返工多、deadline总延期 |
| ③所有团队成员都展现出相应的领导力和人际关系技能 | 成员运用批判性思维和人际关系技能；管理和领导力风格适合项目背景和环境 | 管理和领导力风格适宜性 | 团队成员运用批判性思维解决问题，领导力风格适合项目 | 团队成员不会独立思考，事事请示；领导力风格与项目环境不匹配 |

**指标详解（教材原文展开）**：
- **目标和责任心**：所有项目团队成员都了解愿景和目标；项目团队对项目的可交付物和项目成果承担责任
- **信任与协作程度**：项目团队彼此信任，相互协作
- **适应变化的能力**：项目团队适应不断变化的情况，并在面对挑战时有韧性
- **彼此赋能**：项目团队感到被赋能，同时项目团队对其成员赋能并认可
- **管理和领导力风格适宜性**：项目团队成员运用批判性思维和人际关系技能；项目团队成员的管理和领导力风格适合项目的背景和环境

**考试重点**：论文常要求"制定测量指标"，上表中的指标要会结合实际项目写！

## 四、绩效要点逐个讲

教材原文绩效要点共3个：**项目团队文化**、**高绩效项目团队**、**领导力技能**。

### 要点1：项目团队文化（教材原文）
每个项目团队都会发展出自己的团队文化。团队文化体现的是团队成员怎么工作、怎么互相配合。形成方式有两种：有意识主动形成、非正式自然形成。

每个人都有偏见，可以是有意识或无意识的。公开而透明地对待偏见可建立一种坦诚和信任的文化，这种文化有助于促成共识和协作。项目经理是建立无偏见的项目团队环境的关键。

**确保形成和维护团队文化的7个方法（教材原文）**：
1. **透明**：保持透明有助于识别和分享
2. **诚信**：由职业道德行为和诚实组成
3. **尊重**：尊重每个人及其思维方式、技能以及他们为项目团队的贡献
4. **积极的讨论**：消除误解
5. **支持**：通过解决问题和消除障碍因素向团队成员提供支持，建立支持性文化形成一个信任和协作的环境
6. **勇气**：勇于提出建议、表达异议或尝试新事物有助于形成一种文化
7. **庆祝成功**：当项目达到小阶段目标时，或团队成员做出创新、适应并服务他人和学习等贡献时，要实时庆祝并认可

🏗️ **项目实例**：我建立了三种团队文化：
- **"不追责，只追改进"**：出了问题先解决，再复盘，不骂人——对应"支持"和"积极的讨论"
- **"文档先行"**：所有决策、方案、经验必须写文档，人走知识留——对应"透明"
- **"持续学习"**：每周五下午1小时技术分享会，团队成员轮流讲新技术——对应"庆祝成功"

### 要点2：高绩效项目团队（教材原文）
项目经理通过以下方式打造高绩效团队：

**高绩效项目团队的9个特征（教材原文）**：
1. **开诚布公的沟通**：达成共识、信任和协作的基石
2. **共识**：共享项目的目的及其将带来的收益
3. **共享责任**：团队成员对成果的主人翁意识越强，表现得就越好
4. **信任**：成员相互信任的项目团队愿意付出额外的努力来取得成功
5. **协作**：项目团队相互协作与合作
6. **适应性**：团队能够根据环境和情况调整工作方式，使工作更加有效
7. **韧性**：出现问题或故障时，高绩效项目团队可以快速恢复
8. **赋能**：给项目团队成员赋能，有助于其有权就所采取的工作方式做出决策
9. **认可**：获得认可更有可能继续取得出色绩效

🏗️ **项目实例**：项目进入成熟期后，我的团队表现出高绩效特征：
- **开诚布公的沟通**：每日站会上任何人都可以提出问题，不怕"丢脸"
- **共识**：项目启动时团队共同制定目标，每个人都清楚"我们为什么要做这个项目"
- **共享责任**：数据集成模块出了线上Bug，开发、测试、需求三个人一起复盘，共同找出根因，不追责只追改进
- **信任**：资深开发请假时，其他成员主动接手，不用我安排
- **协作**：开发帮测试写自动化脚本，测试帮需求做用户验收
- **适应性**：客户临时加需求，团队快速调整迭代计划
- **韧性**：服务器宕机后，团队2小时内恢复，没有互相指责
- **赋能**：我安排开发组长参加架构设计培训（能力），明确授权他决定技术方案（权力），他选的方案出了问题我在复盘会上说"这个决策是我授权的，责任我来扛"（信心），方案效果好我在月度会上公开表扬（认可）——赋能=授权+能力建设+心理安全感+认可，缺一不可
- **认可**：每个迭代结束，我在回顾会上公开表扬贡献突出的成员

### 要点3：领导力技能（教材原文）
与领导力相关的特征和活动包括4个方面：

#### 3.1 建立和维护愿景（教材原文）
共同的愿景有助于让大家朝着相同的方向努力。愿景应该具备如下特征：
1. 用强有力的词句或简短的描述对项目做出概括描述
2. 描述可实现的最佳成果
3. 在项目团队成员脑海中形成一幅共同的、有凝聚力的画面
4. 激发人们对实现成果的热情

#### 3.2 批判性思维（教材原文）
项目团队成员可应用批判性思维进行如下工作：
1. 研究和收集无偏见的、均衡的信息
2. 识别、分析和解决问题
3. 识别偏见、未说明的假设以及价值观
4. 辨别语言的使用情况以及对自己和他人的影响
5. 分析数据和证据，以评估论点和观点
6. 观察事件，以识别模式和关系
7. 适当运用归纳、演绎和溯因推理

#### 3.3 激励（教材原文）
激励可以是内在的，也可以是外在的。每个人不止一个激励因素，需了解首要激励因素。

🏗️ **项目实例**：我的领导力体现在三个关键时刻：
- **服务型领导**：开发团队反映"对接老旧系统太痛苦"，我主动协调达梦厂商提供技术支持，帮团队扫除障碍——不是指挥他们干活，而是帮他们解决困难
- **情境领导**：新入职的2名开发需要手把手指导（指令型）；资深开发只需要给目标（授权型）；中间水平的给方向+定期检查（参与型）
- **决策力**：技术选型时，团队在Spring Cloud和Dubbo之间争执不下，我综合评估后拍板用Spring Cloud——领导力不是民主投票，而是在充分听取意见后果断决策

#### 3.4 人际关系技能（教材原文）
人际关系技能包括：**情商**、**决策**、**冲突管理**。

**情商的4个关键领域（教材原文）**：
1. **自我意识**：了解自己的情绪状态
2. **自我管理**：控制自己的情绪反应，不冲动
3. **社交意识**：察觉他人情绪，理解成员需求
4. **社交技能**：管理人际关系，促进团队协作

**解决冲突的方法（教材原文）**：
1. 尊重且坦诚沟通
2. 只聚焦问题本身
3. 关注现在和未来
4. 一起寻找其他方案

🏗️ **项目实例**：需求组和开发组在需求细节上产生冲突，我运用冲突管理方法：
- 先让双方坦诚表达各自观点（尊重且坦诚沟通）
- 把讨论聚焦在"需求到底要实现什么功能"（只聚焦问题本身）
- 不翻旧账，只看接下来怎么解决（关注现在和未来）
- 最终双方一起找到了折中方案（一起寻找其他方案）

## 五、具体怎么落地

### 场景1：新项目组建团队
**做什么**：根据项目需求确定角色（PM、架构师、开发、测试、UI），然后内部抽调+外部招聘。
**写什么文件**：《资源管理计划》《团队章程》，明确团队规则、沟通方式、冲突解决机制。
**用什么工具**：RACI矩阵——谁负责（Responsible）、谁批准（Accountable）、咨询谁（Consulted）、通知谁（Informed）。

### 场景2：团队进入"震荡期"互相看不顺眼
**做什么**：项目经理组织团建（不是吃饭喝酒，是一起解决实际问题），建立共同目标。
**开什么会**：每日站会（15分钟同步进度）、回顾会（复盘哪里好哪里不好）。
**用什么方法**：Tuckman模型——知道这是"震荡期"的正常现象，通过明确角色、建立规则来推进到"规范期"。

### 场景3：核心成员要离职
**做什么**：第一时间一对一谈话，了解原因（钱少了？没成长？受委屈了？），能留的留，留不住赶紧交接+招新人。
**写什么文件**：更新《资源日历》，启动知识转移文档编写。

## 六、和其他绩效域的联动

### 与"干系人绩效域"的联动
**场景**：客户代表（干系人）直接指挥开发团队成员，导致团队不知道听谁的。
**影响**：团队结构被破坏，项目经理权威下降，成员压力增大。
**处理**：明确沟通渠道——所有外部需求通过项目经理，同时让客户参与正式评审会满足其参与感。

### 与"项目工作绩效域"的联动
**场景**：团队工作负荷不均，有人忙死有人闲死。
**影响**：忙的想离职，闲的没成长，整体效率低。
**处理**：通过资源平衡、任务再分配、交叉培训来解决，同时更新项目进度计划。

## 七、涉及的文件和工具

| 文件/工具 | 一句话说明 |
|----------|-----------|
| 资源管理计划 | "用人说明书"，规划什么时候用什么人、用多久 |
| 团队章程 | "团队家规"，约定团队价值观、沟通方式、会议规则 |
| RACI矩阵 | "分工表"，明确谁负责、谁批准、咨询谁、通知谁 |
| 资源日历 | "排班表"，记录每个人什么时候可用、什么时候休假 |
| Tuckman模型 | "团队成长路线图"，形成→震荡→规范→成熟→解散 |

**📐 RACI矩阵简图**：

```
R = Responsible（干活的人）  A = Accountable（背锅/拍板的人）
C = Consulted（被咨询的人）  I = Informed（被通知的人）

         │ PM │ 架构师 │ 开发 │ 测试 │ 业务代表 │
─────────┼────┼───────┼─────┼─────┼─────────│
需求分析 │ A  │   C    │  I  │  I  │    R    │  ← 业务代表干活，PM拍板
架构设计 │ A  │   R    │  C  │  I  │    I    │  ← 架构师干活，PM拍板
编码开发 │ I  │   C    │  R  │  I  │    I    │  ← 开发干活，架构师被咨询
系统测试 │ I  │   I    │  C  │  R  │    I    │  ← 测试干活，开发被咨询
验收确认 │ A  │   I    │  I  │  C  │    R    │  ← 业务代表验收，PM拍板

关键规则：每行必须有且仅有一个A（一件事只能一个人拍板），R可以多个
```

**📐 Tuckman模型简图**：

```
团队绩效
  ↑
  │          ┌──────┐
  │         ╱  成熟  ╲         特征：自组织、高效、默契
  │        ╱   期     ╲        关键词："不用管"
  │    ┌──╱────────────╲──┐
  │    │  规范期          │      特征：规则建立、开始协作
  │    │                  │      关键词："有规矩"
  │    ├──┐              │
  │    │震│              │      特征：冲突、争权、对抗
  │    │荡│              │      关键词："互相看不顺眼" ← 考试最爱考！
  │    │期│              │
  │    └──┘              │
  │  ┌──┐                │      特征：客气、试探、不了解
  │  │形│                │      关键词："互相客气"
  │  │成│                │
  │  │期│                │
  │  └──┘                │
  └──────────────────────┘──→ 时间
                              解散期（项目结束后团队解散）
```

**考试怎么考Tuckman**：
- 选择题：给你一个场景描述（如"团队成员开始争论谁该做什么"），问处于哪个阶段→震荡期
- 案例分析：团队冲突严重，问你原因和对策→震荡期正常现象，需明确角色、建立规则

## 八、容易混淆的概念

| 容易混淆的 | 区别 |
|-----------|------|
| 团队绩效域 vs 资源管理 | 团队绩效域更关注"人"和"团队动力"；资源管理范围更大，包括设备、材料等 |
| 共享责任 vs 集体负责 | 共享责任是积极的文化；集体负责容易变成"法不责众"，需要配合个人考核 |
| 领导力 vs 管理力 | 领导力是"带人"（激励、愿景）；管理力是"管事"（计划、控制） |
| 团队建设 vs 团建活动 | 团队建设是长期过程；团建活动只是手段之一，不是吃顿饭就完事了 |

## 九、常见考试出题点

**选择题挖坑**：
- 团队绩效域3个绩效要点：项目团队文化、高绩效项目团队、领导力技能（不是7个！）
- 团队绩效域3个预期目标：共享责任、建立高绩效团队、所有团队成员都展现出相应的领导力和人际关系技能（不是4个！没有"成员满意"和"能力匹配"）
- 项目团队文化7个方法：透明、诚信、尊重、积极的讨论、支持、勇气、庆祝成功
- 高绩效项目团队9个特征：开诚布公的沟通、共识、共享责任、信任、协作、适应性、韧性、赋能、认可
- 领导力技能4个方面：建立和维护愿景、批判性思维、激励、人际关系技能
