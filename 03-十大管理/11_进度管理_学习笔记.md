# 11 进度管理

> **核对状态**：✅ 已核对（核对日期：2026-05-16）

## 一、一句话理解
进度管理就是"算准时间"——确定项目活动的顺序、估算工期、制定进度计划，并监控实际进度与计划的偏差，确保项目按时完成。

## 二、核心概念与框架

| 过程组 | 过程名称 | 核心作用 |
|--------|---------|---------|
| 规划 | 规划进度管理 | 定规矩——进度怎么规划、怎么控制 |
| 规划 | 定义活动 | 把WBS工作包拆成具体活动 |
| 规划 | 排列活动顺序 | 确定活动之间的依赖关系 |
| 规划 | 估算活动持续时间 | 算每个活动需要多久 |
| 规划 | 制定进度计划 | 整合所有信息，形成进度基准 |
| 监控 | 控制进度 | 监控进度偏差，必要时调整 |

## 三、关键知识点逐个讲

### 知识点0：四种依赖关系（教材原文）
- **强制性依赖（硬逻辑）**：法律、合同或物理性质决定的，不能随意更改（如必须先打地基再建主体）
- **选择性依赖（软逻辑）**：基于最佳实践的，可以调整（可用于快速跟进）
- **外部依赖**：项目团队无法控制的外部因素（如等待政府审批）
- **内部依赖**：项目团队可控的活动间依赖

### 知识点1：关键路径法（CPM）
- **什么意思**：找出决定项目最短工期的活动序列，关键路径上的活动没有浮动时间（总浮动时间=0）。
- **为什么重要**：关键路径决定了项目最短工期，关键路径上的任何延误都会导致项目延期。
- **考试怎么考**：计算题高频考点！给定活动列表和依赖关系，画出网络图，找出关键路径和总工期。

🏗️ **项目实例**：我画了数据治理平台的进度网络图，发现有两条主要路径：
- 路径A：需求调研(4周)→数据标准制定(3周)→数据集成开发(8周)→12家子公司对接(6周)→联调测试(4周) = **25周**
- 路径B：需求调研(4周)→数据安全开发(5周)→安全测评(3周) = **12周**

路径A就是**关键路径**（25周），路径B有13周浮动时间。所以数据集成开发是重中之重，这个模块延期一天，整个项目就延期一天。

### 知识点2：总浮动时间 vs 自由浮动时间
- **什么意思**：总浮动时间是不影响项目完工日期的前提下，活动可以延迟的时间；自由浮动时间是不影响后续活动最早开始时间的前提下，活动可以延迟的时间。
- **为什么重要**：浮动时间是进度优化的重要依据。
- **考试怎么考**：计算题，给定网络图求各活动的总浮动时间和自由浮动时间。

🏗️ **项目实例**：数据安全开发（路径B）的总浮动时间是13周，意味着它可以晚13周开始也不会影响项目总工期。但自由浮动时间取决于后续活动——如果安全测评必须在第20周开始，那安全开发的自由浮动时间就不是13周了。**考试重点**：总浮动时间看项目终点，自由浮动时间看紧后活动。

### 知识点3：赶工 vs 快速跟进
- **什么意思**：赶工是"加资源换时间"（如加班、加人）；快速跟进是"把串行变并行"（如设计和开发同时进行）。
- **为什么重要**：进度延误时需要采取压缩措施。
- **考试怎么考**：选择题考两者的区别和适用场景；案例考进度延误的应对措施。

> **缩短活动工期的6种方法（教材原文，口诀：赶快搞范进质）**：
> 1. 赶工——投入更多资源
> 2. 快速跟进——串行变并行
> 3. 使用高素质资源或经验更丰富的人员
> 4. 减少活动范围或降低活动要求
> 5. 改进方法或技术以提高生产效率
> 6. 加强质量管理，及时发现问题减少返工

🏗️ **项目实例**：数据集成模块开发到第6个月时，发现进度滞后2周。我采取了两种措施：
- **赶工**：从数据安全模块调2名资深开发支援数据集成，加班费增加8万——花钱买时间
- **快速跟进**：把"子公司对接"和"数据清洗"从串行改为并行，先对接数据量少的5家，同时清洗已接入的数据——串行变并行，但增加了数据不一致的风险

最终抢回了2周，项目按期完成。**关键区别**：赶工加成本，快速跟进加风险。

### 知识点4：三点估算
- **什么意思**：用乐观（O）、最可能（M）、悲观（P）三个值来估算工期。
  - **β分布（默认，教材重点）**：期望工期 = (O + 4M + P) / 6
  - **三角分布**：期望工期 = (O + M + P) / 3
- **为什么重要**：单点估算太乐观或太悲观，三点估算更合理。
- **考试怎么考**：计算题，给定三个值算期望工期和标准差。注意题目未指定分布时默认用β分布。

🏗️ **项目实例**：估算"对接一家老旧系统"的工期：
- 乐观：2周（接口文档齐全，顺利对接）
- 最可能：4周（文档不全但能逆向解析）
- 悲观：10周（文档完全没有，对方不配合）

期望工期 = (2 + 4×4 + 10) / 6 = 28/6 ≈ **4.7周**
标准差 = (10 - 2) / 6 = 8/6 ≈ **1.3周**

所以我按5周排计划，但预留了1.3周的缓冲。12家子公司对接总工期按5周/家排，但先易后难分批上线。

### 知识点5：资源平衡 vs 资源平滑
- **什么意思**：资源平衡是"调整进度以解决资源冲突"，可能改变关键路径；资源平滑是"在浮动时间内调整"，不改变关键路径。
- **为什么重要**：资源不足时需要优化资源分配。
- **考试怎么考**：选择题考两者的区别（是否改变关键路径）。

🏗️ **项目实例**：排进度时发现第3个月需要8个开发同时干活，但团队只有5个开发可用——资源超载了。
- **资源平滑**：把非关键路径上的"数据安全开发"推迟2周（它有13周浮动时间），先让开发支援关键路径——**不改变关键路径**
- **资源平衡**：如果平滑不够，就得把部分关键路径上的活动也推迟——**关键路径可能变长，项目延期**

我先试资源平滑，实在不行再资源平衡。

## 五、ITTO串讲（理解式记忆）

**规划进度管理**
- 它在干什么：定规矩——进度怎么规划、怎么控制。
- 核心输出：进度管理计划。

**定义活动**
- 它在干什么：把WBS工作包拆成具体活动（如"开发登录模块"拆成"设计登录页面""编写登录代码""测试登录功能"）。
- 核心输出：活动清单、活动属性、里程碑清单、变更请求。

**排列活动顺序**
- 它在干什么：确定活动之间的依赖关系（FS、SS、FF、SF）。
- 核心输出：项目进度网络图。

**估算活动持续时间**
- 它在干什么：算每个活动需要多久。
- 工具选择：类比估算（参考历史项目）、参数估算（用公式算）、三点估算（乐观/最可能/悲观）。
- 核心输出：持续时间估算。

**制定进度计划**
- 它在干什么：整合所有信息，形成进度基准。
- 关键工具：关键路径法（找关键路径）、资源优化（解决资源冲突）、进度压缩（赶工/快速跟进）。
- 核心输出：进度基准、项目进度计划。

**控制进度**
- 它在干什么：监控进度偏差，必要时调整。
- 核心输出：工作绩效信息、进度预测。

## 七、关键公式（必背！）

| 公式 | 说明 |
|------|------|
| 期望工期 = (O + 4M + P) / 6 | 三点估算（β分布，默认） |
| 期望工期 = (O + M + P) / 3 | 三点估算（三角分布） |
| 标准差 = (P - O) / 6 | 三点估算的标准差 |
| SPI = EV / PV | 进度绩效指数，>1超前，<1滞后 |
| SV = EV - PV | 进度偏差，>0超前，<0滞后 |
| EAC = BAC / CPI | 典型偏差时的完工估算 |
| EAC = AC + (BAC - EV) | 非典型偏差时的完工估算 |
| ETC = EAC - AC | 完工尚需估算 |
| VAC = BAC - EAC | 完工偏差 |

## 八、与其他模块的联动

### 与"成本管理"的联动
进度延误通常导致成本增加（赶工需要加资源），进度压缩也要考虑成本影响。

### 与"范围管理"的联动
范围变更会影响进度，进度压缩可能需要缩小范围。

## 九、常见考试出题点

**选择题挖坑**：
- 关键路径的计算（总浮动时间=0）
- 赶工vs快速跟进的区别
- 资源平衡vs资源平滑的区别（是否改变关键路径）
- 四种依赖关系（FS、SS、FF、SF）

**案例分析怎么考**：
- 计算题：给定活动列表，画网络图，找关键路径，算总工期
- 进度延误分析（原因、影响、应对措施）
- 挣值分析计算（SPI、SV、EAC等）

**论文怎么考**：
- "论信息系统项目的进度管理"——按6个过程写
- 重点写：关键路径分析、进度压缩、挣值分析

## 十、记忆口诀

**进度管理六过程**：
"规定排估制控"——规划、定义、排列、估算、制定、控制

**四种依赖关系**：
"FS最常用，SS同时开始，FF同时结束，SF少见"（开始-完成，如保安接班）

**进度压缩两方法**：
"赶工加资源，快速跟进串改并"

**资源优化两方法**：
"平衡改路径，平滑不改路"
