# 13 质量管理

> **核对状态**：✅ 已核对（核对日期：2026-05-16）

## 一、一句话理解
质量管理就是"确保交付的东西靠谱"——制定质量标准，通过质量保证和质量控制，确保项目产出满足客户要求，不是"能用就行"。

## 二、核心概念与框架

**质量与等级的区别（教材原文）**：
- **质量**：一系列内在特性满足要求的程度。低等级高质量可被接受（如功能简单但稳定的软件）。
- **等级**：对用途相同但技术特性不同的可交付成果的级别分类。高等级低质量不可接受（如功能繁多但缺陷多的软件）。

**五种质量管理水平（教材原文，按有效性递增）**：
1. 客户发现缺陷（代价最大）
2. 控制质量中检测缺陷
3. 通过质量保证纠正过程
4. 将质量融入规划与设计
5. 创建质量文化（最有效）

| 过程组 | 过程名称 | 核心作用 |
|--------|---------|---------|
| 规划 | 规划质量管理 | 定标准——质量目标、验收标准、测试方法 |
| 执行 | 管理质量（质量保证） | 建立信心——过程合规、体系有效 |
| 监控 | 控制质量 | 检查成果——发现缺陷、确保合格 |

**重要区别**：
- 管理质量（质量保证）：关注"过程"——流程对不对、体系健不健全（预防缺陷）
- 控制质量：关注"产品"——成果有没有缺陷、符不符合标准（发现缺陷）

## 三、关键知识点逐个讲

### 知识点1：质量成本（COQ）
- **什么意思**：质量相关的所有成本，包括一致性成本（预防成本+评估成本）和非一致性成本（内部失败成本+外部失败成本）。
- **为什么重要**：预防成本低，失败成本高。前期投入预防，后期少花钱修bug。
- **考试怎么考**：选择题考质量成本的分类；案例考质量投入不足导致后期返工。

**📐 质量成本分类简图**：

```
                        质量成本（COQ）
                       ╱              ╲
              一致性成本              非一致性成本
            （花在"做对"上）         （花在"做错"上）
             ╱        ╲              ╱        ╲
        预防成本    评估成本    内部失败成本  外部失败成本
        （防患未然） （检查把关）  （内部发现）   （外部发现）
           │          │           │           │
        培训        测试        返工        投诉处理
        评审        审计        修复        保修维修
        流程制定    检验        报废        品牌损失
           │          │           │           │
        5万         44万        20万       100万+
           │          │           │           │
           └── 越早投入越省钱 ──┘  └── 越晚发现越费钱 ──┘

记忆口诀："预评内外"——预防、评估、内部失败、外部失败
考试技巧：给你一个场景，判断属于哪种成本——"培训"=预防，"测试"=评估，"返工"=内部失败，"投诉"=外部失败
```

🏗️ **项目实例**：
- **预防成本**：项目初期我安排了代码规范培训（3万）、制定了代码评审流程（2万）——花5万预防
- **评估成本**：测试团队3人×4个月=36万，加上自动化测试工具采购（8万）——花44万评估
- **内部失败成本**：测试阶段发现120个缺陷，返工修复花了约20万——如果前期预防到位，这个数字能减半
- **外部失败成本**：如果缺陷没在上线前修好，上线后用户投诉、紧急修复、品牌损失——可能超过100万

**结论**：花5万预防 >> 花100万救火。考试常考"以下哪个是预防成本/评估成本/失败成本"。

### 知识点2：七种基本质量工具
- **什么意思**：因果图（鱼骨图）、流程图、核查表、帕累托图、直方图、控制图、散点图。
- **为什么重要**：这些工具是质量分析和改进的基础。
- **考试怎么考**：选择题考每种工具的用途（如帕累托图找"关键的少数"、控制图判断过程是否稳定）。

**📐 七种质量工具简图（考试必会！）**：

**① 因果图（鱼骨图）——找根因**：
```
                        接口问题多（结果/鱼头）
                               │
                 ┌─────┬───────┼───────┬─────┐
                 │     │       │       │     │
                人     机      料      法     环
                 │     │       │       │     │
            理解不一致 环境差异  文档不全  无统一  测试环境
                              规范    规范   不稳定
```
用法：把问题放鱼头，从"人机料法环"5个维度找原因

**② 帕累托图——抓重点（80/20法则）**：
```
缺陷数
  60│ ██
  50│ ██
  40│ ██  ██
  30│ ██  ██
  20│ ██  ██  ██
  10│ ██  ██  ██  ██
   0└──────────────────
     接口 数据  UI  性能   ← 按数量从大到小排

累计%  45%  75%  90% 100%   ← 前两类占75%，抓这两类就搞定大部分问题
       └──关键的少数──┘
```
用法：柱状图（数量从大到小）+ 折线（累计百分比），找到"关键的少数"

**③ 控制图——看稳定**：
```
缺陷数
  12│                          ● ← 超出UCL！过程失控！
  10│
   8│─ ─ ─ UCL（控制上限）─ ─ ─ ─ ─ ─ ─ ─ ─
   7│        ●  ●     ●  ●
   6│     ●        ●        ●
   5│─ ─ ─ 中心线（CL）─ ─ ─ ─ ─ ─ ─ ─ ─ ─
   4│  ●     ●     ●     ●
   3│           ●        ●
   2│─ ─ ─ LCL（控制下限）─ ─ ─ ─ ─ ─ ─ ─
   1│
   0└──┬──┬──┬──┬──┬──┬──┬──→ 周数
      1  2  3  4  5  6  7

判异规则（考试必考）：
① 数据点超出控制限 → 过程失控
② 连续7点在中心线同侧 → 过程偏移（七点规则）
③ 连续7点呈趋势上升/下降 → 过程在恶化
④ 连续7点递增或递减 → 过程有趋势
```

**④ 其余4种工具一句话图**：
```
核查表：数数用的表格          │  接口 ▓▓▓▓▓▓ 54个
（先收集数据）                │  数据 ▓▓▓▓ 36个
                              │  UI   ▓▓ 18个
                              │  性能 ▓ 12个

流程图：方框=步骤 菱形=判断    │  子公司数据 →[Kafka]→[清洗]→[入库]
（看流程中的瓶颈）             │                      ↑瓶颈

直方图：看数据分布形态         │  │ ██
（正态？偏态？）               │  │ ██ ██
                              │  │ ██ ██ ██
                              │  └──────── 修复耗时

散点图：两个变量的关系         │      ●
（正相关/负相关/无关）         │    ●   ●
                              │  ●   ●    ← 正相关：评审越多缺陷越少
                              │ ●  ●
                              │● ●
                              │────────── 评审覆盖率
```

🏗️ **项目实例**：系统测试阶段，测试团队发现了120个缺陷，我怎么用七种工具？

| 工具 | 我怎么用的 | 一句话记住 |
|------|-----------|-----------|
| **核查表** | 先用核查表统计：接口问题54个、数据格式问题36个、UI问题18个、性能问题12个 | 先数数，做统计 |
| **帕累托图** | 按数量排序画柱状图，接口问题45%+数据格式30%=75%，这两类是"关键的少数" | 80/20，抓重点 |
| **因果图** | 针对"接口问题多"，画鱼骨图分析：人（双方理解不一致）、机（环境差异）、料（接口文档不全）、法（没有统一规范） | 鱼骨找根因 |
| **流程图** | 画出"数据从子公司→Kafka→清洗→入库"的完整流程，发现清洗环节是瓶颈 | 流程看步骤 |
| **控制图** | 每周跟踪缺陷发现率，画控制图，第7周数据点超出控制上限——过程失控，需要调查原因 | 控制看稳定 |
| **直方图** | 把缺陷按"修复耗时"分组画柱状图，发现大部分缺陷4小时内修复，但有10个超过2天 | 直方看分布 |
| **散点图** | 画"代码评审覆盖率"vs"缺陷密度"的散点图，发现评审覆盖率越高缺陷越少——正相关 | 散点看关系 |

### 知识点3：管理质量 vs 控制质量
- **什么意思**：管理质量是"过程检查"（流程对不对）；控制质量是"产品检查"（成果有没有缺陷）。
- **为什么重要**：两者缺一不可，管理质量预防缺陷，控制质量发现缺陷。
- **考试怎么考**：高频考点！两者的区别和先后顺序。

🏗️ **项目实例**：
- **管理质量（QA）**：我每季度请QA团队做质量审计，检查"代码评审流程有没有执行""测试用例覆盖率够不够""变更有没有走流程"——这是查**过程**
- **控制质量（QC）**：测试团队跑测试用例、做性能测试、检查接口响应时间是否<2秒——这是查**产品**

QA是"防患于未然"，QC是"亡羊补牢"。两者配合：QA确保流程正确减少缺陷产生，QC确保缺陷在交付前被发现。

### 知识点4：控制图
- **什么意思**：用来判断过程是否稳定的工具，有控制上限（UCL）、控制下限（LCL）、规格上限（USL）、规格下限（LSL）。
- **为什么重要**：控制图能提前预警过程失控，防止批量缺陷。
- **考试怎么考**：选择题考控制图的判异规则（七点规则、超出控制限等）。

🏗️ **项目实例**：我每周跟踪"每千行代码缺陷数"，画控制图：
- 中心线 = 5个/千行（历史平均）
- 控制上限UCL = 8个/千行
- 控制下限LCL = 2个/千行

**判异规则**：
1. 数据点超出控制限（如某周突然12个/千行）→ 过程失控，要调查原因
2. **七点规则**：连续7个点在中心线同侧 → 过程偏移，即使没超限也要关注
3. 连续7个点呈上升趋势 → 过程在恶化

第7周数据突然飙到11个/千行，超出UCL，我立即调查发现是新人提交了大量未评审代码，马上加强了代码评审。

## 五、ITTO串讲（理解式记忆）

**规划质量管理**
- 它在干什么：定标准——质量目标、验收标准、测试方法。
- 核心输出：质量管理计划、质量测量指标。

**管理质量（质量保证）**
- 它在干什么：关注"过程"——流程对不对、体系健不健全。
- 核心工具：质量审计（检查流程合规性）、过程分析（找改进点）。
- 核心输出：质量报告。

**控制质量**
- 它在干什么：关注"产品"——成果有没有缺陷、符不符合标准。
- 核心工具：检查、测试、控制图、帕累托图。
- 核心输出：核实的可交付成果（给确认范围用的）。

## 八、质量成本分类

| 类型 | 说明 | 例子 |
|------|------|------|
| 预防成本 | 防止缺陷发生的成本 | 培训、流程设计、质量规划 |
| 评估成本 | 评估成果是否符合要求的成本 | 测试、检查、评审 |
| 内部失败成本 | 交付前发现缺陷的成本 | 返工、废品、停工 |
| 外部失败成本 | 交付后发现缺陷的成本 | 保修、投诉处理、品牌损失 |

**口诀**："预防评估是投资，内部外部是损失"

## 九、与其他模块的联动

### 与"范围管理"的联动
质量标准和验收标准在范围说明书中定义，质量管理确保交付物符合这些标准。

### 与"成本管理"的联动
质量成本是项目成本的一部分，质量投入不足会导致后期返工成本增加。
