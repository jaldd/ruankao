# 01 干系人绩效域

## 一、一句话理解
干系人绩效域就是"搞好人际关系"——识别出所有能影响项目或被项目影响的人，摸清他们的需求和期望，让他们满意、少捣乱、多支持。

## 二、管不好会出什么乱子
1. **项目做着做着突然被喊停**：关键领导没被识别出来，项目快做完了他说"我没同意啊"，直接砍掉。
2. **验收时各种挑刺不签字**：用户代表没参与过程，最后验收说这不行那不行，尾款拿不到。

## 三、预期目标与指标对照（核心！教材原文）

> **🧭 三者关系解读**：绩效要点（你怎么做）→ 预期目标（你要什么结果）→ 指标（你怎么量）。三者是**路径→终点→尺子**的因果链，不是三个独立的东西。下表的"预期目标"就是第四节"绩效要点"要达成的结果，"指标"就是检测目标达没达成的尺子。
>
> **本域映射**：干系人绩效域只有1个要点——"促进干系人参与"（5步循环：识别→分析→排序→参与→监督），这5步做到位了，自然达成4个目标：关系好(①)、认同目标(②)、满意收益(③)、反对者不捣乱(④)。指标就是检测这4个目标达没达成的尺子。
>
> ```
> 5步循环（识别→分析→排序→参与→监督）
>   ├──→ ①与干系人建立高效工作关系  ←尺子：参与连续性
>   ├──→ ②干系人认同项目目标        ←尺子：变更频率
>   ├──→ ③支持者满意并收益          ←尺子：行为+满意度+问题风险
>   └──→ ④反对者未产生负面影响      ←尺子：同上③④共用
> ```

| 预期目标（教材原文） | 大白话解释 | 检查方法/指标（教材原文） | 做得好 | 搞砸了 |
|---------|-----------|-------------|--------|--------|
| ①与干系人建立高效的工作关系 | 和干系人关系好，沟通顺畅 | **干系人参与的连续性**——通过观察、记录方式，对干系人参与的连续性进行衡量 | 干系人登记册定期更新，新干系人能在1周内被识别并纳入管理 | 项目进行到一半突然冒出个"关键领导"说不知道这个项目 |
| ②干系人认同项目目标 | 干系人认可项目方向，不乱改需求 | **变更的频率**——对项目范围、产品需求的大量变更或修改可能表明干系人没有参与进来或与项目目标不一致 | 干系人主动提供资源、主动反馈意见、主动协调关系 | 开会不来、发邮件不回、电话不接，最后关头跳出来反对 |
| ③支持项目的干系人提高了满意度，并从中收益 | 支持的人更满意了，还得到了好处 | **干系人行为**（行为可表明是否满意和支持）、**干系人满意度**（通过调研、访谈和焦点小组方式确定）、**干系人相关问题和风险**（审查问题日志和风险登记册） | 定期满意度调查分数稳定或上升，干系人主动表扬项目团队 | 投诉多、满意度下降、干系人向上级告状 |
| ④反对项目的干系人没有对项目产生负面影响 | 反对的人被管住了，没捣乱 | 同上③④共用检查方法 | 即使有分歧也能通过沟通解决，干系人之间冲突能被有效调解 | 干系人之间公开冲突、互相拆台、把项目当成政治斗争工具 |

**⚠️ 考试重点**：选择题考"预期目标"原文（必须一字不差）；论文考"检查方法/指标"（要会结合实际项目写）！

**绩效要点**：促进干系人参与（5步循环：识别→理解和分析→优先级排序→参与→监督）

## 四、绩效要点逐个讲

### 要点1：识别干系人
**一句话说清**：把跟项目有关的人全部"点名"，一个都不能漏。
**为什么重要**：漏了关键干系人，项目可能突然被喊停。

🏗️ **项目实例**：项目启动前，我拉着核心团队在白板前列出了所有干系人：
- 集团层：总经理、CIO、财务总监、信息中心主任
- 业务部门：工程部、物资部、安全部、财务部
- 子公司层：12家子公司IT负责人
- 项目部层：200+项目的数据录入人员
- 外部：达梦厂商、麒麟厂商、等保测评机构

**差点漏掉的人**：集团审计部！他们后来要求审查数据安全合规性，如果没提前识别，项目可能被审计叫停。识别干系人宁多勿少。

### 要点2：理解和分析干系人
**一句话说清**：摸清每个人的"脾气"——他想要什么、怕什么、权力多大、对项目啥态度。
**为什么重要**：不了解干系人，就无法制定有效的参与策略。

🏗️ **项目实例**：我分析了工程部主任——
- **想要什么**：工程进度实时看板，不用再手工汇总12家子公司的进度报表
- **怕什么**：新系统增加工作量，他手下人不愿意用
- **权力**：中等（他能决定工程部是否配合使用系统）
- **态度**：抵制（觉得新系统是"折腾"）

分析完才知道，他不是反对项目，而是担心增加工作量。我承诺"新系统只减不增工作量"，并设计了"一键导入"功能，他转为中立。

### 要点3：优先级排序
**一句话说清**：人太多管不过来，先伺候"大爷"——权力大、影响大、态度差的优先。
**为什么重要**：资源有限，要把精力花在关键干系人身上。

🏗️ **项目实例**：我用权力/利益方格排序后，发现CIO和信息中心主任是"重点管理"对象。但最需要花精力的是工程部主任——他权力中等但态度抵制，如果不搞定他，工程部不配合使用，项目等于白做。所以优先级不只看权力，还要看"如果不管理他，后果有多严重"。

### 要点4：促进干系人参与
**一句话说清**：把干系人"拉进群"，让他们有参与感，别等项目做完了才通知他们。
**为什么重要**：参与感带来认同感，认同感带来支持。

🏗️ **项目实例**：我让12家子公司IT负责人参与"数据标准制定"的评审会，让他们提意见。虽然多花了2周，但标准制定后他们觉得"这是我们定的"，执行时配合度很高。如果我只在总部闭门制定标准，子公司会觉得"上面强加的"，执行时消极应对。

### 要点5：监督干系人参与
**一句话说清**：定期检查干系人还在不在状态，有没有从"支持"变"反对"了。
**为什么重要**：干系人态度会变，要及时发现并调整策略。

🏗️ **项目实例**：项目第5个月，我注意到物资部主任从"支持"变成了"中立"——他不再主动参加需求评审会了。我赶紧约他一对一沟通，发现他是因为"上次提的需求没被采纳"而不满。我解释了原因并承诺在第二批功能中纳入，他恢复了支持态度。**关键**：干系人态度变化往往有原因，要及时发现、主动沟通。

## 五、具体怎么落地

### 场景1：新项目启动时的干系人识别
**做什么**：开启动会前，项目经理拉着核心团队在白板前 brainstorm ——客户方有谁、公司内部有谁、政府监管部门有谁、竞争对手算不算？全部列出来。
**写什么文件**：《干系人登记册》，包含姓名、职位、联系方式、影响力、态度、需求。
**用什么工具**：权力/利益方格（Power/Interest Grid），把干系人分成四类：重点管理（权力大+利益大）、令其满意（权力大+利益小）、随时告知（权力小+利益大）、最少努力（权力小+利益小）。

### 场景2：干系人态度突然变差
**做什么**：发现某部门经理从"支持"变"中立"甚至"反对"，赶紧一对一喝咖啡聊，了解是不是需求被忽略了。
**写什么文件**：更新《干系人参与计划》，增加对该干系人的沟通频率和内容。
**开什么会**：非正式的"一对一沟通"，正式的"干系人状态评审会"。

### 场景3：项目收尾验收
**做什么**：提前一个月把验收标准发给所有关键干系人确认，别等到最后一天才说"这个我没看过"。
**写什么文件**：《验收报告》需要所有关键干系人签字。

## 六、和其他绩效域的联动

### 与"团队绩效域"的联动
**场景**：项目团队内部有人和外部干系人"关系太铁"，导致干系人绕过项目经理直接指挥团队成员。
**影响**：破坏了团队结构和沟通渠道，团队成员不知道该听谁的。
**处理**：项目经理要明确"所有外部需求必须经过我"，同时把该干系人纳入正式沟通渠道。

### 与"交付绩效域"的联动
**场景**：干系人频繁变更需求，导致交付物一直在改。
**影响**：交付延期、成本超支、团队疲惫。
**处理**：通过干系人管理让关键干系人早期确认需求，后期变更要走正式流程，评估影响后再决定。

## 七、涉及的文件和工具

| 文件/工具 | 一句话说明 |
|----------|-----------|
| 干系人登记册 | "点名册"，记录所有干系人的基本信息和特征 |
| 干系人参与计划 | "攻略手册"，针对不同干系人制定不同的参与策略 |
| 权力/利益方格 | "四象限工具"，帮你看清谁是"大爷"谁是"路人" |
| 干系人参与度评估矩阵 | "态度追踪表"，记录每个干系人当前是支持还是反对 |
| 沟通管理计划 | "聊天指南"，规定跟谁聊、聊什么、多久聊一次 |

**📐 权力/利益方格简图**：

```
                    利益（对项目结果的关注度）
                    低 ←──────────────────→ 高
              ┌──────────────┬──────────────┐
         高   │  令其满意     │  重点管理     │
  权      │  (权力大利益小) │ (权力大利益大) │
  力      │  策略：让他满意  │ 策略：密切管理  │
  （      │  别让他捣乱     │ 主动沟通+满足需求│
  影      │  例：集团审计部  │ 例：CIO、总经理 │
  响      ├──────────────┼──────────────┤
  力      │  最少努力     │  随时告知     │
  ）      │  (权力小利益小) │ (权力小利益大) │
         低   │  策略：花最少精力│ 策略：保持告知   │
              │  例：外围供应商  │ 例：终端用户     │
              └──────────────┴──────────────┘

考试怎么考：
- 选择题：给你一个干系人描述，问该放哪个象限、用什么策略
- 案例分析：干系人管理失败→通常是"重点管理"象限的人没管好
- 易错点：权力大≠利益大！CIO权力大且利益大（重点管理），审计部权力大但利益小（令其满意）
```

参与度5级：不知晓 → 抵制 → 中立 → 支持 → 领导

         不知晓  抵制  中立  支持  领导
          │     │    │    │    │
CIO       │     │    │    │ ██ │ ██ │  ← 当前：领导（期望维持）
工程部主任│     │ ██ │    │    │    │  ← 当前：抵制（需提升到支持）
物资部主任│     │    │ ██ │    │    │  ← 当前：中立（需提升到支持）
终端用户  │ ██ │    │    │    │    │  ← 当前：不知晓（需提升到支持）

策略：识别当前状态 → 设定目标状态 → 制定提升路径
例：工程部主任 抵制→中立→支持（先了解他抵制的原因，再针对性沟通）
```

## 八、容易混淆的概念

| 容易混淆的 | 区别 |
|-----------|------|
| 干系人 vs 用户 | 干系人范围更大，包括所有影响/被影响的人；用户只是干系人中使用产品的那部分 |
| 干系人管理 vs 沟通管理 | 干系人管理是"管人"，沟通管理是"管信息怎么传"；两者有重叠但不等同 |
| 干系人参与 vs 团队管理 | 干系人是项目外的人（或部分项目内高层），团队是项目执行层的人 |

## 九、常见考试出题点

**选择题挖坑**：
- 问"以下哪个不是干系人"——给你挖坑放"项目团队成员"（其实团队成员也是干系人！）
- 权力/利益方格的四类策略对应关系
- 干系人态度从"抵制"变"支持"需要几步

**案例分析怎么考**：
- 给你一个项目失败的案例，让你分析"干系人管理哪里出了问题"
- 常见答案：没识别全干系人、没管理干系人期望、没让干系人参与、沟通不到位

**论文怎么考**：
- "论信息系统项目的干系人管理"——按识别→分析→排序→参与→监督的结构写
- 一定要举具体例子：识别出谁、用什么策略、遇到什么阻力、怎么解决的
- **论文常要求写测量指标**，记住第三节的指标表！

## 十、记忆口诀

**干系人管理五要点**：
"识分排促监"——识别、分析、排序、促进、监督

**权力/利益方格四策略**：
"大权大利重点管，大权利小让他满，小权利大告诉他，小权小利少费力"

**干系人参与度五级**：
"不知抵中支领"——不知晓、抵制、中立、支持、领导
